# MIGRATION_RULES

## Назначение

Перенос старой базы в новую структуру.

---

## Правила

- не считать старые файлы каноном
- разделять:
  - archive (сохранить)
  - inbox (разобрать)
  - удалить (не использовать)
- сначала отделять рабочие материалы от исторического слоя
- не переносить файл в новую структуру без понимания его роли
- спорные и дублирующиеся файлы не относить молча к канону
- если файл важен, но его статус неясен, фиксировать это отдельно

---

## Логика миграции

Для каждого материала определить:

- что это за тип сущности
- актуален ли он для текущей рабочей базы
- является ли он источником истины, рабочим документом, архивом или мусором
- есть ли более новая или более полная версия
- требует ли он переноса, разбора или исключения

---

## Классы решений

### Перенести в рабочую структуру

Если материал:

- актуален
- имеет понятную функцию
- нужен в текущем проекте

### Archive

Если материал:

- больше не является рабочим
- нужен как след проекта
- может понадобиться для сверки, но не для текущей разработки

### Inbox

Если материал:

- потенциально полезен
- не разобран
- смешивает идеи, черновики и канон
- требует ручного решения

### Удалить / не использовать

Если материал:

- явно дублирует более полную версию
- не имеет ценности для проекта
- является техническим шумом

---

## Правила против неоднозначности

- Не объявлять файл каноном только потому, что он выглядит "главным".
- Если есть дубликаты, указывать базовый файл и конфликтующие версии.
- Если файл содержит смешанный материал, фиксировать это как причину отправки в `inbox`.
- Для важных, но неоднозначных файлов указывать риск, если классификация может изменить рабочую базу.

---

## Формат результата

План миграции должен включать:

- источник анализа
- принципы классификации
- переносимые материалы
- archive
- inbox
- удалить / не использовать
- спорные случаи
- риски
- итоговую структуру после миграции
