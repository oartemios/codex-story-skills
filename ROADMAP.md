# Roadmap

Рабочий план развития после первого публичного релиза `v0.1.0`.

## 1. Защитить `main`

Статус: сделано 2026-04-14.

Цель: снизить риск случайно сломать устанавливаемый `main`.

Сделать:

- включить GitHub ruleset или branch protection для `main` — сделано
- требовать успешный `Skills CI` — сделано, обязательный check: `validate`
- запретить force push — сделано
- запретить удаление ветки — сделано
- включить PR-before-merge, если репозиторий начинает принимать внешние PR
- при необходимости требовать up-to-date branch перед merge

Готово, когда:

- `main` нельзя сломать прямым небезопасным изменением — сделано
- `Skills CI` обязателен для merge — сделано

## 2. Включить private vulnerability reporting

Статус: сделано 2026-04-14.

Цель: дать пользователям безопасный канал для security-сообщений.

Сделать:

- включить private vulnerability reporting в настройках GitHub — сделано
- проверить, что `SECURITY.md` соответствует реальному процессу — сделано

Готово, когда:

- у пользователя есть понятный непубличный способ сообщить о security-проблеме — сделано

## 3. Добавить issue templates

Статус: сделано 2026-04-14.

Цель: получать воспроизводимые и полезные issues.

Сделать:

- `bug_report.yml` — сделано
- `wrong_skill.yml` — сделано
- `docs.yml` — сделано
- `feature_request.yml` — сделано

Особенно важен `wrong_skill.yml`, потому что главный риск пакета — routing.

Шаблон `wrong_skill.yml` должен собирать:

- исходный запрос
- ожидаемый skill
- фактически выбранный skill
- почему пользователь считает выбор ошибочным
- структуру проекта, если она нестандартная
- версию пакета или git ref

Готово, когда:

- новые issues содержат достаточно данных для воспроизведения — сделано

## 4. Улучшить установку для внешнего пользователя

Цель: сделать путь установки проще и понятнее.

Сделать:

- решить, оставляем ли основным путь `git clone --branch v0.1.0`
- решить, нужен ли one-liner через `curl`/`bash`
- если one-liner добавляется, явно описать security tradeoff
- обновить `README.md` и `INSTALL.md`

Готово, когда:

- новый пользователь понимает, какой способ установки стабильный
- developer path и user path не смешаны

## 5. Расширить валидатор на top-level docs

Цель: ловить битые ссылки и устаревшие top-level документы до merge.

Сделать:

- расширить `scripts/validate-skills.py`
- проверять markdown-ссылки в:
  - `README.md`
  - `INSTALL.md`
  - `EXAMPLES.md`
  - `SKILLS.md`
  - `TROUBLESHOOTING.md`
  - `CHANGELOG.md`
  - `CONTRIBUTING.md`
  - `SECURITY.md`
- обновить CI, если потребуется

Готово, когда:

- CI падает на битых ссылках в пользовательской документации

## 6. Завести regression prompt suite

Цель: проверять routing и границы skills перед новыми релизами.

Сделать:

- создать `docs/SMOKE_TESTS.md` или `tests/prompts/`
- для каждого кейса указать:
  - запрос
  - ожидаемый skill
  - почему этот skill
  - что считать ошибкой
- покрыть минимум:
  - `project-orchestrator`
  - `continuity-keeper`
  - `writer-assistant`
  - `story-analyst`
  - `project-bootstrap`

Готово, когда:

- перед `v0.1.1` есть ручной regression-набор для проверки routing
