# Каталог Skills

Короткий каталог skills и их границ. Используй его, если непонятно, какой skill нужен для текущей задачи.

## project-orchestrator

Выбирает маршрут, следующий skill и порядок шагов.

Использовать, если:

- непонятно, с чего начать
- задача крупнее одного документа
- нужно понять порядок действий
- нужно проверить готовность к следующему уровню работы
- есть риск перейти к сценам, диагностике или правкам слишком рано

Не использовать как основной skill, если задача уже очевидна:

- собрать канон -> `writer-assistant`
- проверить ритм -> `story-analyst`
- выровнять хаотичную структуру -> `project-bootstrap`
- проверить последствия правок -> `continuity-keeper`

Обычно на выходе:

- ближайший безопасный шаг
- маршрут через несколько skills
- главный блокер
- условия перехода к следующему этапу

Пример:

```text
Хочу довести книгу 1 до рабочего состояния. Что делать сначала?
```

## project-bootstrap

Создает или выравнивает рабочую структуру проекта.

Использовать, если:

- проекта как структуры еще нет
- материалы лежат хаотично
- нужно разнести `canon`, `books`, `characters`, `archive`, `inbox`
- нужна миграция старой базы
- непонятно, где должен лежать материал

Не использовать, если структура уже есть и нужно работать с содержанием. Тогда чаще нужен `writer-assistant`.

Обычно на выходе:

- стартовый каркас проекта
- migration plan
- layout-предложение
- правила, что считать рабочим слоем, архивом и inbox

Пример:

```text
Собери минимальный каркас проекта и предложи, куда разложить текущие материалы.
```

## writer-assistant

Собирает рабочие документы проекта и держит границу между каноном, рабочей версией, идеями и исключенными ветками.

Использовать, если нужно:

- собрать `current-canon.md`
- оформить `system.md`
- собрать персонажа или конфликтную матрицу
- собрать outline, chapter map или scene plan
- привести сырой материал в рабочую форму
- зафиксировать решения после обсуждений

Не использовать как диагностику качества конструкции. Если материал уже собран и нужно найти слабые места, чаще нужен `story-analyst`.

Обычно на выходе:

- рабочий канон
- структура книги или главы
- карточка персонажа
- конфликтная матрица
- scene plan
- уточнение, какие ветки исключены и не должны вернуться

Пример:

```text
Собери current-canon.md по этим материалам и отдели подтвержденное от идей.
```

## story-analyst

Анализирует качество уже существующего материала.

Использовать, если:

- рабочий материал уже есть
- нужно проверить ритм
- нужно найти противоречия
- нужно разобрать арку персонажа
- нужно понять сюжетные риски
- нужен review, а не сборка нового канона

Не использовать, если еще непонятно, что считать рабочим материалом. В таком случае сначала нужен `writer-assistant` или `project-bootstrap`.

Обычно на выходе:

- story diagnostics
- pacing review
- contradiction report
- arc review
- список рисков и следующих действий

Пример:

```text
Проверь ритм книги 1 и укажи, где проседает напряжение.
```

## continuity-keeper

Отслеживает последствия изменений и связи между версиями.

Использовать, если:

- была замена версии
- после правок нужно понять, что сломалось
- нужно сравнить старое и новое состояние
- нужно понять, какие документы синхронизировать
- нужно проверить, не вернулась ли исключенная ветка

Не использовать как основной skill для создания нового материала. Если после проверки нужно собрать обновленный канон, следующим обычно будет `writer-assistant`.

Обычно на выходе:

- continuity report
- список измененных фактов
- список затронутых документов
- риски сломанных связей
- рекомендации по синхронизации

Пример:

```text
Что изменилось после этих правок и что теперь нужно синхронизировать?
```

## rfc-adr-assistant

Пишет, обновляет и проверяет инженерные RFC/ADR.

Поставляется в optional plugin `engineering-addon`.

Использовать, если нужно:

- написать RFC, proposal или design doc
- оформить ADR или decision record
- зафиксировать архитектурное решение
- сравнить варианты и trade-offs
- добавить goals, non-goals, risks, rollout или open questions
- проверить существующий RFC/ADR перед принятием

Не использовать как code review по diff. Если пользователь просит review кода, сначала нужен обычный code-review режим.

Обычно на выходе:

- RFC draft
- ADR draft
- review существующего RFC/ADR
- список alternatives, consequences и open questions

Пример:

```text
Оформи ADR по решению хранить события в append-only таблице.
```

## obsidian-compat

Помогает сопоставить установленный проектный workflow с Obsidian vault, не делая Obsidian обязательным источником истины.

Поставляется в optional plugin `obsidian-addon`.

Использовать, если нужно:

- разложить vault notes по структуре установленного доменного пакета
- понять, какие backlinks, tags или wikilinks являются навигацией, а не каноном
- мигрировать заметки из Obsidian без потери выбранной проектной структуры
- проверить, где vault-навигация расходится с рабочим каноном

Не использовать для обычной работы с каноном, сценами, арками или continuity, если Obsidian не является частью задачи.

Пример:

```text
Помоги сопоставить мой Obsidian vault с рабочей структурой проекта.
```

## Быстрая маршрутизация

- Нет структуры -> `project-bootstrap`
- Есть структура, но нет рабочего канона -> `writer-assistant`
- Есть рабочий материал, нужен анализ качества -> `story-analyst`
- Были правки, нужно понять последствия -> `continuity-keeper`
- Нужно RFC/ADR по инженерному решению -> `rfc-adr-assistant`
- Нужно сопоставить Obsidian vault с проектом -> `obsidian-compat`
- Непонятно, какой пункт выше верный -> `project-orchestrator`

## Типовые цепочки

Новый или хаотичный проект:

```text
project-bootstrap -> writer-assistant -> story-analyst
```

Книга в работе:

```text
writer-assistant -> story-analyst
```

После крупных правок:

```text
continuity-keeper -> writer-assistant -> story-analyst
```

Подготовка к scene-level:

```text
project-orchestrator -> writer-assistant -> story-analyst -> writer-assistant
```

Последняя цепочка означает: сначала проверить готовность уровня, затем укрепить структуру, затем диагностировать риски, и только потом переходить к детальному scene plan.
