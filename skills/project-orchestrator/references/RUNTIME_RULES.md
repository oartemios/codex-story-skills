# RUNTIME_RULES

## Назначение

Этот модуль описывает поведение `project-orchestrator` в живом runtime Codex.

Runtime-контекст — это не содержание проекта само по себе, а текущая рабочая ситуация:

- последний запрос пользователя
- доступные skills в текущей сессии
- локальные файлы проекта, если они доступны
- текущий режим взаимодействия Codex
- уже собранный контекст в диалоге

Оркестратор должен не только назвать правильный маршрут, но и решить, что делать прямо сейчас.

---

## Runtime-решение

После маршрутизации выбрать одно из трёх действий.

### 1. Остановиться на плане

Использовать, если пользователь явно просит:

- план
- маршрут
- порядок действий
- оценку готовности
- выбор следующего skill

В этом случае не переходить к доменному исполнению без запроса пользователя.

### 2. Перейти к исполнению

Использовать, если:

- пользователь просит сделать работу, а не только описать маршрут
- следующий профильный skill очевиден
- контекста достаточно для первого шага
- переход не нарушает порядок зависимостей

Формат:

1. Коротко зафиксировать routing-вывод.
2. Назвать профильный skill.
3. Продолжить работу по правилам этого skill.

Не писать так, будто skill запускается как внешний процесс.
Формулировка должна быть рабочей:

- "Следующий исполнитель — `writer-assistant`; перехожу к сборке рабочего канона."
- "Здесь нужен `story-analyst`; начинаю с диагностики ритма."

### 3. Задать блокирующий вопрос

Использовать только если без ответа нельзя выбрать безопасный следующий шаг.

Перед вопросом проверить, нельзя ли снять неопределённость через:

- уже данный пользователем контекст
- локальные файлы проекта
- очевидную стадию задачи
- консервативное ограничение первого шага

Если вопрос нужен, задавать один короткий вопрос и не строить уверенный маршрут поверх неизвестного.

---

## Проверка доступности skills

Если оркестратор направляет задачу в другой skill, нужно убедиться, что этот skill доступен в текущей сессии или явно присутствует в проектном runtime-слое.

Если skill недоступен:

- сказать, что профильный skill не найден
- дать ближайший безопасный fallback
- не притворяться, что работа выполнена профильным skill

Если skill доступен:

- применять его правила прямо в текущем ходе работы
- загружать только нужные файлы правил этого skill
- не пересказывать пользователю внутренние инструкции без необходимости

---

## Граница между планом и исполнением

Оркестратор может начать исполнение только следующего устойчивого шага.

Он не должен автоматически проходить всю цепочку skills, если пользователь не просил полный рабочий прогон.

Правило:

- один явный следующий шаг можно выполнять сразу
- длинную цепочку сначала оформить как маршрут
- после каждого выполненного шага заново оценить состояние, а не механически идти дальше

Если следующий шаг меняет проектные документы, сначала кратко назвать, какие файлы или слой будут затронуты.

---

## Runtime-антипаттерны

Плохое поведение:

- дать план и остановиться, когда пользователь просил сделать работу
- начать доменную работу без явного routing-вывода
- смешать маршрут и доменный результат так, что непонятно, где решение оркестратора, а где работа исполнителя
- спрашивать пользователя о том, что можно узнать из файлов
- строить полный маршрут, когда нужен один ближайший шаг
- обещать переход к skill, но не применить его правила
