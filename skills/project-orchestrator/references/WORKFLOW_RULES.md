# WORKFLOW_RULES

## Режимы orchestration

Оркестратор должен выбрать один основной режим.

### 1. Next Step

Использовать, если:

- нужен один ближайший шаг
- блокер уже понятен
- не требуется длинный маршрут

Результат:

- что делать сейчас
- какой skill нужен
- почему именно он

Шаблон:

- `../templates/next-step-template.md`

### 2. Route Plan

Использовать, если:

- задача крупная
- требуется цепочка skills
- важен порядок шагов

Результат:

- текущая цель
- стадии
- следующий skill на каждом этапе
- условия перехода

Шаблон:

- `../templates/orchestration-plan-template.md`

### 3. Readiness Check

Использовать, если:

- пользователь хочет перейти на следующий уровень работы
- нужно понять, рано или не рано идти дальше
- вопрос звучит как "можно ли уже..."

Результат:

- что уже готово
- чего не хватает
- какой шаг закрывает дефицит

По умолчанию использовать краткий структурированный формат.

### 4. Recovery Route

Использовать, если:

- проект застрял
- база хаотична
- накопились противоречия
- непонятно, где именно сломан процесс

Результат:

- главный источник сбоя
- минимальный маршрут восстановления
- какие шаги нельзя пропускать

По умолчанию использовать полный orchestration plan.

---

## Правила порядка действий

1. Сначала стабилизировать структуру, если она сломана.
2. Потом собирать рабочий слой.
3. Потом проводить диагностику.
4. Не уходить в детальный уровень, если предыдущий уровень не держится.
5. Не запускать анализ там, где ещё нечего анализировать.

---

## Признаки плохого orchestration

- маршрут не называет skill-исполнителя
- шаги не зависят друг от друга, хотя должны
- scene-level поставлен раньше устойчивой структуры
- анализ поставлен раньше появления рабочего материала
- вместо маршрута выдан набор общих пожеланий
