# Смоук-Тесты

Ручной regression-набор для проверки routing и границ skills перед релизами.

## Как использовать

1. Собрать plugin bundle через `python3 .codex-dev/scripts/build-plugins.py`.
2. Установить нужный plugin через Codex plugin management или, для локального source-теста, через `.codex-dev/scripts/sync-to-codex.sh`.
3. Перезапустить Codex, чтобы он перечитал plugins или skills.
4. Запустить запросы ниже в тестовом художественном проекте или в минимальной папке с похожей структурой.
5. Для каждого кейса проверить:
   - выбранный skill
   - не смешана ли роль skill с соседним доменом
   - не предложен ли преждевременный следующий уровень работы

Не использовать приватный manuscript content в regression-отчетах. Если кейс требует контекста, использовать минимальный синтетический пример.

## Кейсы

### ORCH-01

Запрос:

```text
Хочу проработать книгу 1
```

Ожидаемый skill: `project-orchestrator`

Почему: запрос крупнее одного доменного действия и требует маршрута по книге.

Ошибка:

- сразу начать писать канон без определения стадии
- сразу проводить диагностику
- выдать общий совет без следующего skill

### ORCH-02

Запрос:

```text
Что мне делать сначала, а что потом
```

Ожидаемый skill: `project-orchestrator`

Почему: пользователь просит порядок действий, а не доменное исполнение.

Ошибка:

- перейти к `writer-assistant` без routing-вывода
- спрашивать несколько уточнений, если можно дать ближайший безопасный next step

### BOOT-01

Запрос:

```text
Собери минимальный каркас проекта
```

Ожидаемый skill: `project-bootstrap`

Почему: задача про стартовую структуру и раскладку рабочей базы.

Ошибка:

- собирать канон вместо структуры
- проводить анализ качества материала
- не различать `archive` и `inbox`

### WRIT-01

Запрос:

```text
Собери рабочий канон
```

Ожидаемый skill: `writer-assistant`

Почему: задача про создание или обновление рабочего канонического документа.

Ошибка:

- проводить story diagnostics вместо сборки канона
- возвращать исключенные ветки без явного основания
- смешивать подтвержденное, идеи и старые версии

### WRIT-02

Запрос:

```text
Собери current-canon.md по этим материалам
```

Ожидаемый skill: `writer-assistant`

Почему: запрос явно на рабочий канон и файл `current-canon.md`.

Ошибка:

- выбрать `story-analyst`
- выбрать `continuity-keeper`, если нет сравнения версий или правок
- не отделить канон от гипотез

### ANALYST-01

Запрос:

```text
Проверь ритм книги
```

Ожидаемый skill: `story-analyst`

Почему: задача про качество конструкции и pacing review.

Ошибка:

- собирать outline вместо анализа
- переписывать сцены вместо диагностики
- игнорировать отсутствие рабочего материала, если анализировать нечего

### ANALYST-02

Запрос:

```text
Разбери противоречия в книге 1
```

Ожидаемый skill: `story-analyst`

Почему: запрос на диагностику согласованности и противоречий.

Ошибка:

- отправить задачу в `writer-assistant` без причины
- сгладить противоречия вместо фиксации риска
- переписать канон вместо отчета

### CONT-01

Запрос:

```text
Что изменилось после этих правок
```

Ожидаемый skill: `continuity-keeper`

Почему: задача про сравнение изменений и последствия правок.

Ошибка:

- начать писать новую версию материала
- проводить общий story review без change tracking
- не назвать затронутые связи

### CONT-02

Запрос:

```text
Что теперь нужно синхронизировать после замены версии
```

Ожидаемый skill: `continuity-keeper`

Почему: пользователь просит список зависимых документов и синхронизацию после замены версии.

Ошибка:

- выбрать `project-bootstrap`
- выбрать `writer-assistant` до фиксации изменившихся связей
- не отделить измененные факты от неизмененных

### DEV-01

Запрос:

```text
Оформи ADR по этому архитектурному решению
```

Ожидаемый skill: `rfc-adr-assistant`

Почему: запрос на architecture decision record, а не на художественный канон или сюжетную диагностику.

Ошибка:

- выбрать `writer-assistant`
- писать RFC вместо ADR без причины
- ставить статус `Accepted`, если пользователь не сказал, что решение принято

### DEV-02

Запрос:

```text
Напиши RFC для миграции API
```

Ожидаемый skill: `rfc-adr-assistant`

Почему: запрос на proposal/design doc для технического изменения.

Ошибка:

- выбрать `project-orchestrator` без необходимости
- пропустить alternatives, non-goals или rollout
- закрыть open questions догадками

### BOUNDARY-01

Запрос:

```text
Можно ли уже переходить к scene plan для книги 1?
```

Ожидаемый skill: `project-orchestrator`

Почему: это readiness check между chapter-level и scene-level.

Ошибка:

- сразу строить scene plan без проверки chapter-level устойчивости
- отправить в `story-analyst`, если вопрос не про качество уже собранного материала, а про порядок перехода
- не назвать условия готовности
